Kaggle CMI Sensor Data: Transformation Assessment

Can the CMI Detect Behavior with Sensor Data Dataset Be Transformed
for MediaPipe Model Training?

Abstract

This report assesses whether the Kaggle Child Mind Institute (CMI) Detect Behavior with Sensor Data dataset can be meaningfully transformed into training data for a MediaPipe-based computer vision BFRB (Body-Focused Repetitive Behavior) detection model. After thorough technical analysis, the conclusion is that the dataset cannot be meaningfully transformed. The sensor modalities between the wrist-worn Kaggle device and a webcam-based MediaPipe system are fundamentally incompatible — a physics-level mismatch rather than an engineering limitation.

1. Introduction and Background

The BFRB Detection Web App project (ENEB453) uses a webcam with MediaPipe Holistic to detect body-focused repetitive behaviors in real time. During initial planning, the Kaggle CMI competition dataset was evaluated as a potential training source. This report documents the outcome of that evaluation and explains in technical detail why the two data modalities are incompatible.

2. What the Kaggle CMI Dataset Contains

The CMI Detect Behavior with Sensor Data competition dataset was collected from a custom wrist-worn wearable device called Helios. Subjects wore the bracelet while performing labeled BFRB gestures and control activities. The dataset contains a total of 574,945 rows across 81 subjects and 18 gesture classes, with 341 sensor columns per row.

4. Why Transformation Is Not Possible: Technical Analysis

For the Kaggle dataset to be used to train the MediaPipe model, each sensor reading would need to be converted into an equivalent MediaPipe landmark feature. The following subsections examine each sensor type and explain why each mapping fails.

4.1 Thermal Sensors — No Camera Equivalent

The Kaggle device includes 5 thermal sensors that measure actual skin temperature at the point of contact. When a hand touches the face, thermal readings rise from approximately 28°C to 33°C. Feature importance analysis of the Kaggle model confirms that thermal readings account for approximately 27% of total model importance — making them the single most informative sensor group.

A webcam captures visible light, not infrared radiation or thermal energy. There is no proxy, approximation, or workaround for this missing signal. This represents a complete loss of the dataset's most discriminative feature class.

4.2 Time-of-Flight Sensors — Different Physical Basis

The five ToF sensors emit infrared light and measure the physical distance in millimeters from the wrist to nearby surfaces, providing a 5×64 spatial depth grid. This accounts for approximately 45% of model importance.

MediaPipe provides hand-to-face distances in normalized 2D image coordinates (0–1), which differ from physical millimeter measurements in several fundamental ways:

  • Distances are affected by camera position (user proximity changes pixel ratios)
  • The relationship between normalized coordinates and physical millimeters is non-linear and varies with camera focal length, lens distortion, and user position
  • Depth information is unreliable in MediaPipe's z-axis estimation
  • Only 7 landmark distance pairs are available versus 320 ToF values — a 97.8% reduction in information

4.3 Accelerometer — Incompatible Scale

Wrist velocity can be approximated from MediaPipe by tracking the hand landmark frame-to-frame. However, the Kaggle dataset measures actual acceleration in g-force (an inertial measurement), while MediaPipe provides pixel displacement per frame (a visual measurement). A model trained on physical acceleration values cannot apply those learned decision boundaries to pixel-space velocity values. The relationship between the two is non-linear and varies with camera setup.

4.4 Quaternion Rotation — Different Reference Frames

Quaternions represent 3D wrist orientation relative to gravity, measured by a gyroscope. Palm orientation can be estimated from MediaPipe hand landmarks, but the reference frames are completely different: the IMU uses a world gravity frame while MediaPipe uses a 2D camera image frame. The values are qualitatively similar but quantitatively incompatible.

5. Version 1 Transformation Attempt and Results

A cross-domain feature transformation was implemented and tested as the first approach. Sixteen features were engineered to bridge the Kaggle and MediaPipe domains:

Kaggle Sensor → Engineered Feature → MediaPipe Proxy ──────────────────────────────────────────────────────────────────────── Accelerometer (3-axis) → acc_mag_mean/std/max → Wrist velocity from hand[0] Δ Quaternion (4-axis) → rot_speed_mean → Palm direction vector Thermal (5 sensors) → thm_mean, thm_max → Hand-face overlap fraction ToF grid 1 (64 values) → tof1_min_mean, tof1_mean → Wrist-to-nose distance ToF grid 2 (64 values) → tof2_min_mean, tof2_mean → Index-to-mouth distance ToF grid 3 (64 values) → tof3_min_mean, tof3_mean → Thumb-to-cheek distance ToF grid 4 (64 values) → tof4_min_mean, tof4_mean → Palm-to-forehead distance ToF grid 5 (64 values) → tof5_min_mean, tof5_mean → Wrist-to-chin distance

Training results on the Kaggle test set:

  • Accuracy: 77%
  • AUC-ROC: 0.837
  • Top feature importances: thermal readings (27%), ToF grids (45%), accelerometer (20%)

Why the model failed on real webcam data:

  1. The model learned that thermal signal rise indicates BFRB contact. The webcam has no thermal signal.
  2. The model learned ToF distance patterns in millimeters. MediaPipe distances are in 0–1 normalized pixel ratios.
  3. MinMax normalization does not resolve this — it normalizes within each domain, but the decision boundaries that separate BFRB from non-BFRB are defined in the source sensor domain's physical scale.
  4. This constitutes a zero-shot cross-domain transfer: the model never observed a single MediaPipe feature during training.

7. Why Creative Workarounds Also Fail

7.1 Scaling ToF mm to Pixel Ratios

The relationship between physical distance and pixel distance depends on camera focal length, sensor size, user distance from camera, and lens distortion. These parameters vary per user and per session, making a single universal scaling function impossible.

7.2 Using Labels Without Features

The Kaggle dataset does contain labeled BFRB gestures. Using that label knowledge to inform rule design is valid — and is exactly what the v3 rule-based engine does. However, using labels without features is not "transforming the data for model training."

7.3 Temporal Pattern Learning

BFRB temporal patterns such as dwell duration and frequency are already encoded in the rule-based engine's timers. The Kaggle data provides no new information beyond what is available from published clinical BFRB literature.

7.4 Domain Adaptation Techniques

Domain adaptation methods such as DANN or CycleGAN require paired samples from both domains or at minimum features in the same dimensionality and modality. Wearable sensor data and camera landmark data have no structural correspondence that can be aligned.

8. Chosen Alternative: Rule-Based Engine on MediaPipe Landmarks

Rather than forcing an incompatible cross-domain transfer, the project implemented a detection engine that operates natively on MediaPipe landmarks. This approach includes:

  • A 4-state machine (IDLE → PROXIMITY → DWELL → BFRB) using face-width-normalized Euclidean distances
  • A pinch-pull engine for detecting hair pulling and skin picking via 2-finger pinch and pull-away velocity
  • v3 additions: mouth aperture filtering, extended finger count, micro-motion jitter analysis, palm orientation, dual-signal confidence scoring, and negative detectors for eating and resting behaviors

This approach eliminates the domain gap entirely and achieves real-time detection at 30 fps with significantly fewer false positives than the cross-domain model approach.

9. Conclusion

The Kaggle CMI dataset is valuable for understanding BFRB behaviors and validating that wearable-based detection is feasible. However, the data was collected from physical sensors — accelerometer, gyroscope, thermal, and time-of-flight — that measure fundamentally different quantities than what a webcam and MediaPipe system can observe.

The three critical blockers that make transformation impossible are:

  1. Thermal data has no camera equivalent. Seventy-two percent of the model's top features depend on sensors that do not exist in a webcam-based system.
  2. Physical units cannot be mapped to normalized pixel coordinates without per-user camera calibration data that is unavailable at runtime.
  3. Zero-shot cross-domain transfer requires structural feature alignment that does not exist between wearable sensor data and camera landmark data.

The correct engineering decision — which has been implemented — is to use MediaPipe's native landmark data directly with rule-based heuristics, leveraging behavioral knowledge from BFRB research without attempting to force incompatible sensor data through a machine learning model.
