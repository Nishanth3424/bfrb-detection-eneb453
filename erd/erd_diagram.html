🔗 Relationships & Foreign Keys

bfrb_landmark_rules.bfrb_type_id → bfrb_types.id MANY-TO-ONE  |  ON DELETE CASCADE

detections.bfrb_type_id → bfrb_types.id MANY-TO-ONE  |  ON DELETE RESTRICT

✅ Normalization to 3rd Normal Form (3NF)

1NF — First Normal Form ✓
All columns hold atomic (indivisible) values. No column contains lists or nested structures. Every row is uniquely identified by its primary key. Detection landmark rules are stored in their own rows in bfrb_landmark_rules rather than as a JSON blob inside bfrb_types — this achieves atomicity.
2NF — Second Normal Form ✓
All tables use a single-column integer primary key (id), so there are no composite keys and therefore no partial dependencies possible. Every non-key attribute depends on the whole primary key.
3NF — Third Normal Form ✓
No transitive dependencies exist. For example, detections stores bfrb_type_id (not the behavior name directly) — the name is looked up via the FK relationship. This means a behavior name change only needs to happen in one place (bfrb_types.name), not across every detection row. No non-key attribute determines another non-key attribute in any table.
